BelTech STD-003
AI Generation Specification for SystemVerilog UVM Testbenches
Profile: Small-to-Medium FPGA IP Projects
1. Purpose

This specification defines the required and recommended structure, rules, templates, and generation behavior for an AI system creating a SystemVerilog UVM testbench for FPGA IP development.

This standard is intended for:

small to medium sized FPGA projects
block-level and IP-level verification
Vivado/XSim-based development flows
requirements-driven verification with practical reuse

This standard is not intended to force large-program overhead onto small projects. Features that are typically justified only for large SoC verification efforts are intentionally excluded unless specifically requested.

2. Scope

This specification applies to AI-generated UVM testbenches that verify:

custom RTL IP
register-based blocks
protocol adapters
control/status interfaces
small datapath and interface components

This specification assumes:

one DUT per testbench
one primary top-level simulation module
one or more DUT interfaces
one primary UVM environment
requirement-to-testbench traceability
transaction-based stimulus and checking
3. Compliance Levels
3.1 Required

These items shall be generated for every compliant testbench.

3.2 Strongly Recommended

These items should be generated by default unless the user explicitly requests a minimal bench.

3.3 Optional

These items may be generated when justified by DUT complexity.

The AI shall generate the following files for every compliant UVM testbench:

build.tcl

This expands your earlier minimum set by separating reusable base sequence and base test behavior into their own files and by making coverage a dedicated subscriber component rather than a loose placeholder. Your current spec already establishes the core required structure and compile model that this revision builds on.

5. Strongly Recommended File Set

For small-to-medium FPGA IP projects, the AI should also generate:

dut_tb_types_pkg.sv
dut_tb_params_pkg.sv
dut_tb_utils_pkg.sv

dut_agent_cfg.sv
dut_env_cfg.sv

seq_R<REQ_ID>_<name>.sv
dut_regression_test.sv

dut_sva.sv
dut_req_traceability.csv
5.1 Purpose of these files

dut_tb_types_pkg.sv
Defines enums, typedefs, shared transaction-related types, and protocol state types.

dut_tb_params_pkg.sv
Defines shared constants, timing parameters, address ranges, protocol values, default timeouts, and scoreboard limits.

dut_tb_utils_pkg.sv
Defines helper functions, formatting routines, compare helpers, and common verification utility tasks.

dut_agent_cfg.sv
Defines agent configuration such as active/passive mode, protocol options, timing knobs, and interface role selection.

dut_env_cfg.sv
Defines environment-level enables such as scoreboard enable, coverage enable, assertion enable, timeout limits, and verbosity knobs.

seq_R<REQ_ID>_<name>.sv
Defines requirement-specific stimulus sequences for traceable verification.

dut_regression_test.sv
Defines a reusable regression-style test for requirement-driven sequence execution.

dut_sva.sv
Defines protocol or timing assertions for cycle-accurate requirements.

dut_req_traceability.csv
Defines requirement traceability outside the source code for review and closure.

6. Optional File Set

The following files may be generated only when justified:

dut_ref_model.sv
dut_predictor.sv
dut_bind.sv
dut_reg_model_pkg.sv
run_regression.tcl

These are optional for the intended project size profile and should not be generated by default unless the DUT or requirements warrant them.

7. Naming Rules

The AI shall follow these naming rules:

Artifact	Naming Rule
Top-level simulation module	tb_top
Interface file	dut_if.sv
Package file	dut_pkg.sv
Sequence item	dut_seq_item
Base sequence	dut_base_seq
Requirement sequence	seq_R<REQ_ID>_<name>
Sequencer	dut_sequencer
Driver	dut_driver
Monitor	dut_monitor
Agent	dut_agent
Environment	dut_env
Scoreboard	dut_scoreboard
Coverage subscriber	dut_coverage_subscriber
Base test	dut_base_test
Smoke test	dut_smoke_test
Regression test	dut_regression_test
Assertion group	dut_sva

The top-level simulation module shall be named tb_top, and the simulator shall elaborate work.tb_top. Test selection shall use +UVM_TESTNAME=<test_name>. These are already correctly locked down in your current standard and should remain unchanged.

8. Package and Compile Rules
8.1 Package Rule

All class-based UVM source files shall be included within dut_pkg.sv.

Individual class files shall not be compiled directly.

8.2 Compile Order Rule

The build system shall compile in the following order:

dut_if.sv
dut_pkg.sv
tb_top.sv
8.3 Vivado/XSim Rule

The AI shall generate a build.tcl that:

compiles interface first
compiles package second
compiles top last
elaborates work.tb_top
runs with +UVM_TESTNAME=<test_name>

These compile and entry rules are foundational and should remain mandatory.

9. Sequence Item Rule

A sequence item file shall always be present.

All driver stimulus shall originate from sequence items.

The driver shall not invent transactions internally.

The sequence item shall serve as the transaction-level data model linking:

requirements
stimulus
checking
coverage

This remains a core requirement from your current spec and should continue to be enforced.

10. Type Safety Rule

The following are forbidden:

uvm_driver #(uvm_sequence_item)
uvm_sequencer #(uvm_sequence_item)

The following are required:

uvm_driver #(dut_seq_item)
uvm_sequencer #(dut_seq_item)

All typed TLM connections shall use concrete transaction types.

11. Constants, Types, and Shared Parameters
11.1 Shared Constants Rule

All shared protocol constants, timing values, address limits, default delays, and response codes shall be defined in dut_tb_params_pkg.sv.

Magic numbers shall not be duplicated across driver, monitor, scoreboard, sequences, or assertions.

11.2 Shared Types Rule

All shared enums, typedefs, state encodings, and protocol role definitions shall be defined in dut_tb_types_pkg.sv.

11.3 Shared Utility Rule

Common formatting, comparison, and helper routines should be placed in dut_tb_utils_pkg.sv.

This is one of the biggest practical improvements over a bare-minimum bench because it prevents duplicated constants and keeps AI-generated files aligned.

12. Configuration Object Rules
12.1 Agent Configuration

dut_agent_cfg.sv should define:

active/passive mode
protocol role
timing options
agent enable flags
protocol-specific knobs

12.2 Environment Configuration

dut_env_cfg.sv should define:

scoreboard enable 
scoreboard_enable knob - provided so directed tests can suppress shadow checking intentionally — this is a UVMF pattern that prevents false failures in negative testing.
coverage enable
assertion enable
timeout limits
error policy
verbosity level


12.3 Configuration Delivery

Configuration objects should be passed through uvm_config_db.

Hardcoded operating modes should be avoided unless the bench is intentionally minimal.

13. Connectivity Rules
13.1 Driver to Sequencer

The agent shall connect the driver and sequencer:

drv.seq_item_port.connect(seqr.seq_item_export);

13.2 Monitor to Scoreboard

The monitor shall define a typed analysis port:

uvm_analysis_port #(dut_seq_item) ap;

The environment shall connect monitor output to the scoreboard or to intermediate analysis infrastructure.

13.3 Coverage Subscriber Connectivity

The environment should connect the monitor analysis port to the coverage subscriber.

13.4 Analysis FIFO Recommendation

If the DUT produces output that is temporally separated from the input stimulus, the scoreboard should use uvm_tlm_analysis_fifo rather than relying only on direct write() callbacks.

These connectivity requirements are extensions of the mandatory monitor-analysis and sequencer-driver rules already present in your current specification.

14. Virtual Interface Rules

All verification components that access DUT signals shall:

declare a virtual dut_if vif
retrieve the interface using uvm_config_db

The top module shall assign the interface into the UVM configuration database before run_test().

Direct DUT signal access from class-based components without a virtual interface is forbidden.

15. Sequence Rules
15.1 Base Sequence

The AI shall generate dut_base_seq.sv.

This file shall contain:

common sequence startup behavior
optional reset wait helpers
reusable utility tasks
common transaction creation methods

15.2 Requirement Sequences

The AI should generate one or more requirement-specific sequence files named:

seq_R<REQ_ID>_<name>.sv

Each requirement-specific sequence should extend dut_base_seq.

15.3 Smoke Sequence Requirement

At least one smoke-style sequence shall be present and shall be runnable from the smoke test.

15.4 Interface vs. Environment/Testbench Sequences
All sequences in a single-agent environment are interface sequences by default.
Higher tiers apply when multi-agent coordination is needed.

16. Test Rules
16.1 Base Test

The AI shall generate dut_base_test.sv.

This file shall:

instantiate the environment
apply environment configuration
define common startup and objection policy
16.2 Smoke Test

The AI shall generate dut_smoke_test.sv.

This file shall:

extend dut_base_test
start at least one valid sequence
execute a basic legal DUT transaction flow
16.3 Regression Test

The AI should generate dut_regression_test.sv when there are multiple requirement-specific sequences.

17. Scoreboard Rules

The scoreboard shall implement requirement-based checks.

The scoreboard shall not exist merely as an empty placeholder.

If the DUT behavior is simple, the scoreboard may compare observed transactions directly.

If the DUT behavior requires transformation or stateful expectation tracking, the AI should generate a predictor or reference model.

All scoreboard errors should identify the requirement ID when applicable.

18. Coverage Rules
18.1 Coverage Component

Coverage should be implemented in dut_coverage_subscriber.sv as a uvm_subscriber #(dut_seq_item) or equivalent typed coverage component.

18.2 Requirement Coverage

Each requirement shall have at least one identifiable coverage element.

18.3 Coverage Naming

Coverage groups and coverpoints should include the requirement ID when tied to a specific requirement.

Your current standard already requires requirement-linked coverage. This revision mainly formalizes the structure of the coverage component and makes it more reusable.

19. Assertion Rules

dut_sva.sv should be generated by default for small-to-medium IP whenever the DUT has:

handshake timing
reset timing
latency limits
illegal state transitions
protocol temporal rules

Assertions should be used for cycle-accurate or temporal requirements.

Scoreboards should be used for transaction-level functional correctness.

20. Reset and Clock Rules

The top module shall define explicit clock and reset generation.

Reset behavior shall not be left implicit.

If reset sequencing affects verification legality, the AI shall ensure sequences and monitors wait for reset release before operating.

21. Requirements Traceability Rules

Every requirement shall map to:

at least one sequence or stimulus source
at least one checker or assertion
at least one coverage point or coverage group

The AI should generate dut_req_traceability.csv with columns similar to:

Requirement ID, Description, Sequence, Scoreboard Check, Assertion, Coverage, Status

This extends the traceability rules already present in your current document into a more reviewable artifact.

22. Build Script Rules

The AI shall generate build.tcl.

The script shall:

compile with SystemVerilog enabled
include the UVM library
compile interface, package, then top
elaborate work.tb_top
run the requested UVM test
fail clearly on compile or elaboration errors

For Vivado/XSim, the build script shall be compatible with small project command-line execution.

24. Completion Criteria

A generated testbench is compliant only if it:

compiles without error
elaborates work.tb_top
runs at least one valid UVM test
uses typed sequence items
connects driver, sequencer, monitor, scoreboard, and coverage correctly
contains at least one meaningful scoreboard check
contains at least one meaningful coverage definition
supports requirement traceability
includes a valid build.tcl
25. Default Generation Policy for FPGA IP

Unless the user requests otherwise, the AI should generate:

a single interface
one agent
one environment
one scoreboard
one coverage subscriber
one base test
one smoke test
one base sequence
one or more requirement sequences
shared params/types packages
config objects
a Tcl build script
assertion file when timing or protocol behavior exists

Unless explicitly requested, the AI should not generate:

full SoC-level register model infrastructure
multi-agent fabric-level environments
excessive factory override complexity
large regression harnesses beyond the needs of the IP
26. Summary

This revised standard is intended to be strict enough for AI code generation, but still practical for the size of project you are targeting.

It preserves the strongest parts of your existing spec:

tb_top as the simulation entry
required sequence items
strict typing
package-based compile model
monitor/scoreboard connectivity
requirements traceability
Vivado-compatible execution

It adds the pieces that are highly recommended for real-world IP verification:

shared constants and types
config objects
base sequence and base test
coverage subscriber
assertions
traceability artifact
practical build structure
