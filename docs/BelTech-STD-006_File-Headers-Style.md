# BelTech-STD-006 — File Header Style Standard

**Document number:** BelTech-STD-006
**Revision:** 1
**Date:** 2026-04-14
**Author:** S. Belvin, BelTech Systems LLC
**Status:** Active

---

## 1. Purpose and Scope

This document defines the required file header format for all source files
created or maintained within BelTech Systems LLC projects. A compliant header
provides consistent identification, traceability, implementation status, and
change history in a machine-readable format.

This standard applies to:

- VHDL source files (`.vhd`)
- SystemVerilog source files (`.sv`, `.svh`)
- Python source files (`.py`)
- Tcl script files (`.tcl`)

Files generated by pssgen must carry a compliant header on every output file.
The `scaffold_gen.py` module is the authoritative generator for SystemVerilog
and Tcl headers in the pssgen pipeline.

Related standards: BelTech-STD-001 (HDL Style), BelTech-STD-003 (UVM Testbench
Style).

---

## 2. Common Requirements

### 2.1 Required fields

All headers, regardless of file type, must contain the following fields in the
order shown:

| Field | Description |
|---|---|
| **File** | Filename only — no path |
| **Project** | Project name and one-line description |
| **Brief** | One or two sentence functional description of this file |
| **Document** | Specification reference and revision (e.g. `BelTech-STD-003 Rev 1`) |
| **Standard** | Language standard and/or tool version |
| **Functional blocks** | Named blocks with brief purpose; one entry per block |
| **Dependencies** | Direct compile-time or include dependencies |
| **Portability** | Tool or language portability constraints |
| **Impl. status** | One of the vocabulary values defined in Section 7 |
| **History** | Ordered list of changes, oldest first |

### 2.2 Separator lines

The header opens and closes with a full-width separator line. The separator
style depends on the file type:

| File type | Comment prefix | Separator body |
|---|---|---|
| VHDL | `-- ` | `=` × 61 (total line: 64 characters) |
| SystemVerilog / Tcl | `// ` or `# ` | `=` × 62 (total line: 65 characters) |

### 2.3 Alignment

Field labels are left-aligned in a fixed-width column. The colon appears after
the label at a consistent column position, followed by one or more spaces and
then the field value. Continuation lines for multi-line values are indented to
align with the start of the first value character on the previous line.

### 2.4 History entry format

Each history entry follows this format:

```
YYYY-MM-DD  Author  Description of change
```

For pssgen-generated files, the author field is `pssgen` and the description
is `Generated by scaffold_gen.generate_uvm_tb()`.

## 3. VHDL File Header

VHDL files use `--` comment syntax. The separator line is `--` followed by
61 `=` characters.

### 3.1 Template

```vhdl
-- =============================================================
-- File       : filename.vhd
-- Project    : project name and description
-- Brief      : one or two sentence functional description
-- Document   : specification reference and revision
-- Standard   : VHDL-YYYY (IEEE 1076-YYYY)
--
-- Functional blocks and processes:
--   BLOCK_NAME   : PROCESS_NAME — purpose description
--   BLOCK_NAME   : PROCESS_NAME — purpose description
--
-- Dependencies:
--   library.package_name
--   library.package_name
--
-- Portability    : notes on tool or language constraints
-- Impl. status   : stub / partial / complete / validated
--
-- History:
--   YYYY-MM-DD  Author  Description
-- =============================================================
```

### 3.2 Reference implementation

The VHDL DUT in `ip/buffered_axi_lite_uart/vhdl/buffered_axi_lite_uart.vhd`
is the reference implementation for this standard. Its header is reproduced
here verbatim:

```vhdl
-- =============================================================
-- File       : buffered_axi_lite_uart.vhd
-- Project    : pssgen Example IP — Buffered AXI-Lite UART
-- Brief      : AXI-Lite slave UART with NCO baud generator,
--              16-deep TX/RX FIFOs, and 8 interrupt sources.
-- Document   : BALU-RS-001 Rev 0.4
-- Standard   : VHDL-2008 (IEEE 1076-2008)
--
-- Functional blocks and processes:
--   PARAM_CHECK    : Elaboration-time generic assertions
--   NCO_BAUD       : NCO_ACCUM_p — NCO accumulator, baud pulse
--   AXI_WRITE_CTRL : AXI_AW_LATCH_p, AXI_W_LATCH_p,
--                    AXI_WRITE_RESP_p — AXI-Lite write channel
--   AXI_READ_CTRL  : AXI_READ_p — AXI-Lite read channel
--   REG_BLOCK      : REG_WRITE_p — register file (reset + write decode)
--   TX_FIFO        : TX_FIFO_p — transmit FIFO
--   RX_FIFO        : RX_FIFO_p — receive FIFO
--   TX_ENGINE      : TX_ENGINE_p — UART transmit shift register
--   RX_ENGINE      : RX_ENGINE_p — UART receive shift register
--   INT_CTRL       : INT_CTRL_p — interrupt sticky flags, IRQ
--   TIMEOUT_CTRL   : TIMEOUT_p — receive timeout counter
--   STATUS_MUX     : STATUS_p — process(all) STATUS register assembly;
--                    drives status_word_s; AXI_READ_p muxes into rdata_s
--
-- Dependencies:
--   ieee.std_logic_1164
--   ieee.numeric_std
--   ieee.math_real (elaboration only — for baud reset calc)
--
-- Portability    : ieee.math_real used at elaboration only; no synthesis impact
-- Impl. status   : complete — parity TX/RX, loopback, and stop-bits
--                  implemented; ghdl and iverilog verified
--
-- History:
--   2026-04-07  S. Belton  Initial entity + architecture stub
--   2026-04-10  S. Belton  Reset branches implemented, STATUS_p complete
--                          with live hardware signal assembly
--   2026-04-10  S. Belton  Full architecture implementation
--   2026-04-11  S. Belvin  fix: mark parity RX — remove check per UART-FF-008
-- =============================================================
```

### 3.3 Notes

- The `Brief` field may span two lines. Continuation lines align with the first
  character of the value on the preceding line.
- Functional blocks and processes are listed together under one heading. Each
  block entry identifies one or more process names and their purpose.
- When `ieee.math_real` is used, the Portability field must note that it is
  used at elaboration only.

## 4. SystemVerilog File Header

SystemVerilog files use `//` comment syntax. The separator line is `//`
followed by a space and 62 `=` characters.

### 4.1 Template

```systemverilog
// ==============================================================
// File       : filename.sv
// Project    : pssgen — AI-Driven PSS + UVM + C Testbench Generator
// Copyright  : Copyright (c) 2026 BelTech Systems LLC
// License    : MIT License — see LICENSE for details
// ==============================================================
// Brief      : one or two sentence functional description
// Document   : BelTech-STD-003 Rev 1
// Standard   : SystemVerilog IEEE 1800-2017 / UVM 1.2
//
// Functional blocks:
//   block_name     : purpose description
//   block_name     : purpose description
//
// Dependencies:
//   dependency_name
//   dependency_name
//
// Portability    : Simulator-independent / Vivado-Questa-Icarus
// Impl. status   : generated / stub / partial / complete / validated
//
// History:
//   YYYY-MM-DD  Author/Tool  Description
// ==============================================================
```

### 4.2 Field values for pssgen-generated UVM files

| Field | Required value |
|---|---|
| **Project** | `pssgen — AI-Driven PSS + UVM + C Testbench Generator` |
| **Copyright** | `Copyright (c) 2026 BelTech Systems LLC` |
| **License** | `MIT License — see LICENSE for details` |
| **Document** | `BelTech-STD-003 Rev 1` |
| **Standard** | `SystemVerilog IEEE 1800-2017 / UVM 1.2` |
| **Impl. status** | `generated` for structural files; `stub` for COV sequence files |
| **History author** | `pssgen` |
| **History description** | `Generated by scaffold_gen.generate_uvm_tb()` |

### 4.3 Portability field conventions

| Value | Meaning |
|---|---|
| `Simulator-independent` | Compiles with Vivado XSIM, Questa, and Icarus without modification |
| `Vivado/Questa/Icarus` | Compiles with all three; may use simulator-specific pragmas in comments |
| `Vivado 2024.x / XSIM` | Tcl/build scripts — simulator-specific |

### 4.4 Example: generated interface file

```systemverilog
// ==============================================================
// File       : buffered_axi_lite_uart_if.sv
// Project    : pssgen — AI-Driven PSS + UVM + C Testbench Generator
// Copyright  : Copyright (c) 2026 BelTech Systems LLC
// License    : MIT License — see LICENSE for details
// ==============================================================
// Brief      : AXI-Lite interface bundle for buffered_axi_lite_uart
//              DUT connections.
// Document   : BelTech-STD-003 Rev 1
// Standard   : SystemVerilog IEEE 1800-2017 / UVM 1.2
//
// Functional blocks:
//   Interface signals  : All DUT ports as logic signals
//
// Dependencies:
//   UVM 1.2
//
// Portability    : Simulator-independent
// Impl. status   : generated
//
// History:
//   2026-04-14  pssgen  Generated by scaffold_gen.generate_uvm_tb()
// ==============================================================
```

## 5. Tcl File Header

Tcl scripts use `#` comment syntax. The separator line is `#` followed by
a space and 62 `=` characters. The `Standard` field is replaced by `Tool`
because Tcl scripts are simulator-specific.

### 5.1 Template

```tcl
# ==============================================================
# File       : filename.tcl
# Project    : pssgen — AI-Driven PSS + UVM + C Testbench Generator
# Copyright  : Copyright (c) 2026 BelTech Systems LLC
# License    : MIT License — see LICENSE for details
# ==============================================================
# Brief      : one or two sentence description of what this script does
# Document   : BelTech-STD-003 Rev 1
# Tool       : Vivado 2024.x / XSIM
#
# Usage:
#   vivado -mode tcl -source filename.tcl
#
# Impl. status   : generated / reviewed / validated
#
# History:
#   YYYY-MM-DD  Author/Tool  Description
# ==============================================================
```

### 5.2 Field values for pssgen-generated build scripts

| Field | Required value |
|---|---|
| **Project** | `pssgen — AI-Driven PSS + UVM + C Testbench Generator` |
| **Copyright** | `Copyright (c) 2026 BelTech Systems LLC` |
| **License** | `MIT License — see LICENSE for details` |
| **Document** | `BelTech-STD-003 Rev 1` |
| **Tool** | `Vivado 2024.x / XSIM` (for Vivado build scripts) |
| **Impl. status** | `generated` |

### 5.3 Example: generated build script

```tcl
# ==============================================================
# File       : build.tcl
# Project    : pssgen — AI-Driven PSS + UVM + C Testbench Generator
# Copyright  : Copyright (c) 2026 BelTech Systems LLC
# License    : MIT License — see LICENSE for details
# ==============================================================
# Brief      : buffered_axi_lite_uart Vivado/XSIM build and run script —
#              three-file compile model per STD-003.
# Document   : BelTech-STD-003 Rev 1
# Tool       : Vivado 2024.x / XSIM
#
# Usage:
#   vivado -mode tcl -source build.tcl
#
# Impl. status   : generated
#
# History:
#   2026-04-14  pssgen  Generated by scaffold_gen.generate_uvm_tb()
# ==============================================================
```

## 6. Python File Header

Python files use `#` comment syntax with a distinct layout that accommodates
the module-level docstring pattern. The format is already established in the
pssgen codebase and is codified here.

### 6.1 Template

```python
# ===========================================================
# FILE:         path/filename.py
# PROJECT:      pssgen — AI-Driven PSS + UVM + C Testbench Generator
# COPYRIGHT:    Copyright (c) 2026 BelTech Systems LLC
# LICENSE:      MIT License — see LICENSE file for details
# ===========================================================
#
# DESCRIPTION:
#   Two to four sentence description of what this module does,
#   what layer it belongs to, and what problem it solves.
#
# LAYER:        N — layer name
# PHASE:        vX / phase-identifier
#
# FUNCTIONS:
#   function_name(args)
#     One to two line description.
#   function_name(args)
#     One to two line description.
#
# DEPENDENCIES:
#   Standard library:  module, module
#   Third-party:       package
#   Internal:          module, module
#
# HISTORY:
#   vX.Y  YYYY-MM-DD  Author  Description
#
# ===========================================================
```

### 6.2 Notes

- The separator uses 59 `=` characters (not 62), for a total line width of 62
  characters including the leading `# `.
- The `LAYER` field uses the five-layer numbering from the pssgen architecture:
  1 — parser, 2 — ir, 3 — agents, 4 — checkers, 5 — emitters.
- The module-level docstring immediately follows the closing `# ====` line.

## 7. Implementation Status Vocabulary

The `Impl. status` field uses a controlled vocabulary. Values map to VPR
Disposition column values where applicable.

| Value | Meaning | VPR Disposition |
|---|---|---|
| `generated` | Produced by pssgen `scaffold_gen`; structurally correct but content not yet reviewed by an engineer. Suitable for simulation but not for tape-out or compliance submission. | `GENERATED` |
| `stub` | Skeleton only; body intentionally not implemented. Requires engineer implementation before simulation use. Used for COV sequence files. | `GENERATED` (body not implemented) |
| `partial` | Some functional blocks implemented; not yet complete. Engineer work in progress. | — |
| `complete` | All functional blocks implemented; not yet simulation-verified. | — |
| `validated` | Simulation-verified; passes regression with zero errors and zero `UVM_ERROR` messages. | `COVERED` or `PASSING` |

### 7.1 Promotion path

The expected lifecycle for a pssgen-generated file:

```
generated  →  partial  →  complete  →  validated
```

For COV sequence stubs:

```
stub  →  partial  →  complete  →  validated
```

Engineers update the `Impl. status` field in the header when they promote a
file. pssgen never modifies a file that already exists at the target path
(never-overwrite contract, CLAUDE.md Rule 5).

## 8. pssgen Generation Rules

When `scaffold_gen.generate_uvm_tb()` generates a file, the following rules
apply automatically:

1. `Impl. status` is set to `generated` for all structural files:
   `_if.sv`, `_pkg.sv`, `_seqr.sv`, `_agent.sv`, `_env.sv`, `_test.sv`,
   `_base_seq.sv`, `_smoke_seq.sv`, `_regression_test.sv`, `_driver.sv`,
   `_monitor.sv`, `_scoreboard.sv`, `_coverage_subscriber.sv`, `tb_top.sv`,
   `build.tcl`.

2. `Impl. status` is set to `stub` for COV sequence files
   (`seq_RCOV*.sv` — body not implemented; see VPR for guidance).

3. The History entry is:
   ```
   YYYY-MM-DD  pssgen  Generated by scaffold_gen.generate_uvm_tb()
   ```
   where `YYYY-MM-DD` is the date of generation.

4. The `Document` field references `BelTech-STD-003 Rev 1`.

5. The `Standard` field for `.sv` files is `SystemVerilog IEEE 1800-2017 / UVM 1.2`.

6. The `Tool` field for Tcl files is `Vivado 2024.x / XSIM`.
